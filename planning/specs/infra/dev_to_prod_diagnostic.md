High-risk / major blockers
- No deploy artifacts or automation: ops/deploy and ops/scripts are empty, and there is no Dockerfile or infra-as-code; only local docker-compose with bind mounts [docker-compose.yml](/Users/adamyu/Desktop/Optimized_Life/baseline/fieldpro_pilot/docker-compose.yml).
- Persistent services depend on localhost networking assumptions (DB host, OSRM default URL, MinIO endpoint) baked into code and env; will break when containers run on separate hosts or behind load balancers [backend/src/db.ts](/Users/adamyu/Desktop/Optimized_Life/baseline/fieldpro_pilot/backend/src/db.ts), [backend/src/osrmClient.ts](/Users/adamyu/Desktop/Optimized_Life/baseline/fieldpro_pilot/backend/src/osrmClient.ts), [backend/src/s3Client.ts](/Users/adamyu/Desktop/Optimized_Life/baseline/fieldpro_pilot/backend/src/s3Client.ts).
- Secrets and IDs stored in plaintext repo env files and committed keys (MapTiler, Azure IDs) [backend/.env](/Users/adamyu/Desktop/Optimized_Life/baseline/fieldpro_pilot/backend/.env), [frontend/.env.local](/Users/adamyu/Desktop/Optimized_Life/baseline/fieldpro_pilot/frontend/.env.local).
- No migration orchestration; schema changes require manual script execution tied to a single file [backend/src/run_migration_washed_can.ts](/Users/adamyu/Desktop/Optimized_Life/baseline/fieldpro_pilot/backend/src/run_migration_washed_can.ts), risking drift from production.
- Observability and backup gaps: no metrics/log forwarding, no automated DB or object backups, and no health checks wired to infra [ops/](/Users/adamyu/Desktop/Optimized_Life/baseline/fieldpro_pilot/ops), [db_dumps/](/Users/adamyu/Desktop/Optimized_Life/baseline/fieldpro_pilot/db_dumps), [data/minio](/Users/adamyu/Desktop/Optimized_Life/baseline/fieldpro_pilot/data/minio).

Likely breakage when moving off local Docker
- Backend will fail to reach Postgres because host is hard-coded to `localhost` instead of a service DNS name [backend/src/db.ts](/Users/adamyu/Desktop/Optimized_Life/baseline/fieldpro_pilot/backend/src/db.ts).
- OSRM client defaults to `http://localhost:5005` while compose exposes OSRM on host port 5005 but uses container name `fieldpro_osrm`; moving to cloud without that port mapping will break [backend/src/osrmClient.ts](/Users/adamyu/Desktop/Optimized_Life/baseline/fieldpro_pilot/backend/src/osrmClient.ts), [docker-compose.yml](/Users/adamyu/Desktop/Optimized_Life/baseline/fieldpro_pilot/docker-compose.yml).
- MinIO endpoint in env uses localhost and HTTP; any TLS-enforced S3-compatible service or different hostname will invalidate presigned URLs [backend/.env](/Users/adamyu/Desktop/Optimized_Life/baseline/fieldpro_pilot/backend/.env), [backend/src/s3Client.ts](/Users/adamyu/Desktop/Optimized_Life/baseline/fieldpro_pilot/backend/src/s3Client.ts).
- Frontend proxy and MSAL redirect hard-coded to localhost ports; production hostnames will cause login redirect failures and API 404s [frontend/vite.config.ts](/Users/adamyu/Desktop/Optimized_Life/baseline/fieldpro_pilot/frontend/vite.config.ts), [frontend/src/msalConfig.ts](/Users/adamyu/Desktop/Optimized_Life/baseline/fieldpro_pilot/frontend/src/msalConfig.ts).
- Bind-mounted data directories (`./data/db`, `./data/minio`, `./data/osrm`) assume writable local filesystem; in stateless containers these paths will be missing unless replaced with managed volumes [docker-compose.yml](/Users/adamyu/Desktop/Optimized_Life/baseline/fieldpro_pilot/docker-compose.yml).

Summary
- Overall migration difficulty: 8/10. The core app is simple, but infra, config, and operability are largely dev-only and must be stood up from scratch.
- Top 5 blockers: (1) lack of deploy artifacts/automation, (2) hard-coded localhost dependencies (DB/OSRM/MinIO), (3) missing production-grade config and secrets management, (4) absent observability/backup story, (5) schema migration tooling is ad hoc.
- First fixes to ease production: standardize env config for DB/OSRM/MinIO with non-local defaults; add Dockerfiles and a production compose/Kubernetes chart plus secrets delivery; lock down CORS and rotate committed keys; introduce migration runner and CI pipeline; add basic monitoring/backup procedures (liveness endpoints into health checks, Postgres snapshots, object storage versioning).
