# 2026-05-15 — S1-7 gap fix: audit log entry + .env.example documentation

## What changed

- `backend/src/constants.ts` (new) — defines `SYSTEM_ACTOR_OID = 'system:baseline-automated'`,
  the synthetic actor identifier used when audit entries are written by automated
  scripts rather than by a real user session; overrideable via `SYSTEM_ACTOR_OID` env var
- `backend/src/scripts/populateEamBridge.ts` — imports `writeAuditLog` and
  `SYSTEM_ACTOR_OID`; awaits a `writeAuditLog` call at the end of each successful
  `populate()` run with action `admin.eam_bridge_populate` and detail
  `{ rows_written, run_at, route_run_ids_processed_count }`; uses `await` (not
  fire-and-forget) because the script calls `pool.end()` immediately after `populate()`
  returns — the fire-and-forget pattern is unsafe in that context
- `backend/src/middleware/auditActions.ts` — added `'admin.eam_bridge_populate'`
  to `AUDIT_KNOWN_ACTIONS`
- `backend/.env.example` — added S1-7 section documenting the populate script's
  invocation method, DB dependency, and the optional `SYSTEM_ACTOR_OID` override

## Why

- S1-7 audit-report (planning/security/S1-AUDIT-REPORT.md) flagged two unchecked
  done-criteria: audit log entry per populate run, and env var documentation
- Without the audit entry, EAM bridge scheduled runs were invisible to the audit
  trail — an Admin querying the audit log could not confirm the script had run
  or how many rows it processed
- Without .env.example documentation, operators had no reference for how to
  schedule or configure the populate script
