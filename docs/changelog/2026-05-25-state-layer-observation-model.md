# 2026-05-25 — Canonical state layer: two-axis observation model (condition / action / measurement / presence), absence-as-counted-signal, no stored arrival state

## What changed
- `planning/architecture/CANONICAL_STATE_LAYER_DESIGN.md` refined to a two-axis observation model. Status remains **TARGET DESIGN — PENDING §9 VERIFICATION**; banner moved to the top of the doc and restated.
- **Obs kinds reset to four:** `condition`, `action`, `measurement`, `presence`. The previous `state` kind was renamed to `condition` for clarity. Action is now a first-class kind — an intervention performed (e.g. `washed_pad`, `picked_up_litter`) — and is explicitly **not** subject to ok/not_ok normalization.
- **norm_status is kind-conditional.** Condition and measurement rows carry a graded `ok | not_ok | unknown`. Presence rows carry NULL (row existence is the signal — they are problem-flags). Action rows carry NULL status and populate a new `intervention` normalized column instead. The `core.observations` DDL, the registry `ok_rule` rules, the §4.2 normalizer pseudocode, the §4.3 view, and the §6 write path were all updated to match.
- **Invariants rewritten/added** in §2:
  - **#3 condition/action independence** — they are separate axes and never collapsed into one status (this was the original defect: a chronically-dirty-but-maintained asset was indistinguishable from a clean one when both axes were folded into a single ok/not_ok).
  - **#4 absence is a counted signal** — the previous "absence is data, but presence is better" was replaced. An unchecked component within a completed visit is the absence of an observation, measured against the registry's declared possible types and anchored by a `core.visits` row. Never force-filled with default ok/not_ok.
  - **#5 no default or inferred arrival state** — no observation is auto-created on visit start. A default arrival state would be an un-asserted guess (the exact work-order-inference weakness this product rejects).
  - **#6 store the facts, derive the transition** — arrival→departure change is never stored as a row or paired before/after record; it is derived from rate/trend of condition + action observations.
  - **#7 capture-UI-is-the-contract** — the schema must never require a value the live capture surface does not collect. The capture surface produces actions, problem-flags, one stop-level spot check, one measurement, and evidence; therefore no arrival condition is stored.
- **§3.5 added — the spot check is a stop-level positive condition anchor.** It is `kind=condition`, `norm_status=ok`, scope=stop. It is a **completed servicing visit** with outcome "assessed, no work needed" — not a non-service event and not an intervention. It is what makes component-level silence safe to read as benign (§4.4).
- **§4.4 added — Absence as a counted signal.** Specifies that absence becomes legible only against (a) a visit anchor in `core.visits` and (b) the registry's declared full set of possible types for the asset type. Includes the conceptual LEFT JOIN shape (visit → registry possible types → observations, NULL = silence, counted). States that intelligence reads rate/trend of asserted facts against visit-completeness anchors, never a per-visit arrival value.
- **§5.1 added — Adapter is bidirectional.** On write: org vocabulary → canonical facts. On read: canonical facts → org-vocabulary projections (operational report in KCM language; normalized signals for intelligence; analyst MV surface in org vocabulary). Notes that the MV layer is the same physical artifact serving both intelligence performance and analyst querying.
- **§8a added — Three queryable tiers** exposed to analysts: raw canonical (`core.observations`/`core.visits`), normalized (`v_observation_normalized`), and aggregated MVs (KCM-vocabulary rollups). All three queryable; MV tier answers ~95% of questions; raw stays accessible so analysts retain ownership and intelligence remains verifiable against counts.
- **§8b added — Daily operations report filter** (recorded as a downstream consumer example, not a build spec): operations surfaces only (i) stops with a `not_ok` condition assertion and (ii) stops on a completed route with no `core.visits` anchor. Routine cleanings (action observations) and spot checks are intelligence accumulation, not operational alerting. The only non-service visit outcome is `skipped` with `reason_code='safety'`.
- **§7 worked example** updated to use the four kinds and to show action and spot-check rows alongside conditions/measurements/presence.
- **§9 updated:** the paired-before/after open question is **resolved** (superseded by the new model — two facts plus derived transition). A new open question replaces it: do all 25 live `core.observation_type_registry` rows map cleanly to exactly one of the four kinds? Existing-row migration question (Q4) was extended to call out the same reclassification check.

## Why
- The previous single-status model (`norm_status` covering all kinds) collapsed two genuinely independent axes — what condition was found vs. what action was taken — into one column. That made the system structurally incapable of distinguishing a chronically-dirty-but-maintained asset (lots of action observations, lots of `not_ok` conditions) from a genuinely clean one (few of either). The degradation signal the product depends on lived in the difference between those axes; collapsing them destroyed it.
- A default or auto-populated arrival condition would be a worker-independent guess — the same work-order-inference weakness this product rejects in its core thesis. The capture UI never asks the worker to rate arrival state; the schema must reflect that, or it carries values the field did not produce.
- Silence is a real signal in this product (an unchecked component on a completed visit *means something*). Modeling it as a missing-row pattern measured against a registry-declared denominator and a visit-presence anchor keeps it logically certain rather than inferred.
- Aligning the schema to the actual capture surface (actions, problem-flags, one stop-level anchor, one measurement, evidence) ensures the canonical layer never requires a value the field cannot produce — and that absence of a row carries the same meaning everywhere.
