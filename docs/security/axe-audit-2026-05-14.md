### Pattern 1 — Tailwind `text-gray-300` on White Backgrounds

- **Rule**: `color-contrast`
- **WCAG**: 1.4.3 (AA) — 4.5:1 minimum for normal text
- **Severity**: serious
- **Surfaces**: Login (1 node), Control Center (4 nodes), Lead Routes (1 incomplete candidate)
- **Total confirmed nodes**: 5 | **Potential with incomplete resolved**: 7
- **Root cause**: A single Tailwind utility class, `text-gray-300` (`#d1d5dc`), is used for empty-state messages, secondary labels, and version text. Against a white background it yields ratio 1.47:1 (Login badge uses `#94a3b8` = ratio 2.56:1 — different token, same pattern).
- **Remediation**: Replace `text-gray-300` with `text-gray-500` (`#6b7280`, ratio 4.6:1) or `text-gray-600` for all secondary/empty-state text. The Login version badge uses an inline style — change its `color` value to `#64748b` (slate-500) or darker.
- **Scope**: do not introduce new color tokens. Use only existing Tailwind defaults (`gray-500`, `gray-600`, `slate-500`).

---

### Pattern 2 — Unfocusable Horizontal-Scroll Containers

- **Rule**: `scrollable-region-focusable`
- **WCAG**: 2.1.1 (AA) — keyboard accessible
- **Severity**: serious
- **Surfaces**: Lead Routes (2 nodes), Control Center (1 node)
- **Total nodes**: 3
- **Root cause**: `<div class="overflow-x-auto">` wrappers around data tables lack `tabindex="0"`. Keyboard users cannot scroll horizontally via arrow keys. Screen reader virtual cursor on VoiceOver/NVDA cannot reach table content inside an unfocusable container.
- **Remediation**: Add `tabindex="0"` to every `overflow-x-auto` div that wraps a table. No ARIA role needed — keyboard focusability alone satisfies the rule. Verify with keyboard tab-through after fix.

## Part C — Manual Accessibility Audit (2026-05-14)

### Check 1 — Focus Traps on Modal Dialogs

**Findings**: All 5 custom modal dialogs lacked `role="dialog"`, `aria-modal="true"`, and `aria-labelledby`. Without these attributes, screen readers do not announce the dialog context when a modal opens, and do not restrict virtual cursor navigation to modal content.

**Affected components**:
- `ConfirmDialog.tsx` — skip-stop confirmation
- `ImagePreviewModal.tsx` — photo preview
- `ConflictResolutionModal.tsx` — offline conflict management
- `StopDetail.tsx` Safety Modal (inline)
- `StopDetail.tsx` Infra Modal (inline)

**Fixes applied**: Added `role="dialog"` + `aria-modal="true"` + `aria-labelledby` (with matching `id` on heading) to all 5 modals.

**Not addressed (follow-up required)**: Focus management JavaScript — moving focus to first interactive element on open, containing Tab within the dialog, returning focus to the trigger on close. This requires `useEffect` hooks per dialog and constitutes a feature addition beyond the S1-9 scope. Each dialog currently relies on the user pressing Escape or clicking the backdrop, both of which work. Flagged for S1-10 or next a11y sprint.

### Check 2 — Logical Focus Order

**Result**: PASS. All 6 surfaces use sequential DOM layout. No CSS absolute positioning reorders interactive elements relative to DOM order. Modals append at end of their component output but are visually elevated via `fixed inset-0`.

### Check 3 — Touch Target Size (WCAG 2.5.5 AAA)

**Findings**:

1. **"← Back to Route" button** (`StopDetail.tsx` line 422 — not-started early return): missing `min-h-[44px]`. **Fixed**: added `min-h-[44px] flex items-center`.

2. **Photo thumbnail remove button** (`StopDetail.tsx` photo strip): `w-5 h-5` (20px square). Cannot be expanded to 44×44px without visually covering the thumbnail it overlays. **Flagged as product decision — not fixed.** This button is only reachable after a user selects a photo for upload (pre-upload state, not primary flow).

All other buttons across all 6 surfaces use `min-h-[44px]` consistently.

### Check 4 — Viewport Reflow at 320px (WCAG 1.4.10 AA)

**Result**: PASS.
- `UlLayout` uses `px-4` (16px per side), giving 288px content at 320px viewport — no horizontal overflow.
- All tables use `overflow-x-auto` with `tabIndex={0}` — scroll accessible at 320px.
- `grid-cols-2` in Safety/Infra modals: ~136px per column at 320px — tight but no content clips or overlaps.
- No fixed-width elements exceeding 320px outside of scrollable containers.

### Check 5 — Color-Only State Indication (WCAG 1.3.3 / 1.4.1)

**Findings**:

1. **Trash volume selector** (`StopDetail.tsx`): 0-4 buttons indicate selected state only via `bg-gray-100`/`text-gray-900` vs `bg-white`/`text-gray-500` + `shadow-inner`. No programmatic state exposed to assistive technology. **Fixed**: added `aria-pressed={checklist.trashVolume === val}` to each button.

2. **`OpsBadge` status variants**: Renders the status string as text content (value prop is always visible text). Color is supplementary. **PASS**.

3. **`StopListItem` status badges**: Color + text label ("Pending", "Done", etc.). **PASS**.

4. **`StopChecklist` items**: Native `<input type="checkbox">` — state conveyed via checked/unchecked attribute. **PASS**.

5. **Progress bar in `RouteHeader`**: Supplemented by visible "X of Y complete" text. **PASS**.

### Check 6 — VoiceOver Spot-Check

**Not automatable** — requires macOS VoiceOver (VO + arrow navigation) running in a real browser session. Key surfaces to verify manually before production:

- **UL Stop List**: Announce `<ul>` list with N items; each `<li>` announces stop number + address + status badge.
- **ConfirmDialog**: With the ARIA fixes applied, VO should now announce "Report Safety Concern, dialog" when the Safety Modal opens. Verify focus moves to the dialog or that VO virtual cursor enters correctly.
- **ConflictResolutionModal**: Verify "Stops needing attention, dialog" announced on open.
